# TV_Server

Локальний медіасервер, розроблений для забезпечення потокового відтворення торентів та агрегованого пошуку. Ідеально підходить для використання з додатком **Lampa** на Smart TV, Android TV, або ПК.

## 🚀 Архітектура та склад

Цей стек розгорнуто в ізольованому середовищі за допомогою **Docker Compose** і складається з двох основних компонентів:

1. **TorrServer** (порт `8090`)
   - **Призначення:** Потокове відтворення торентів (Streaming) без необхідності попереднього завантаження всього файлу на жорсткий диск.
   - **Оптимізація:** Скрипт `configure.ps1` вмикає розширений кеш в оперативній пам'яті (2 ГБ) та збільшує ліміт P2P-з'єднань (до 1000) для максимальної утилізації ширини каналу (до 300+ Мбіт/с). Значення налаштовуються у файлі `.env`.
   - Дані кешуються виключно в ОЗП і видаляються після перегляду.

2. **Jackett** (порт `9117`)
   - **Призначення:** Уніфікований проксі-сервер для пошуку по торент-трекерам. Перекладає пошукові запити від Lampa у запити до конкретних сайтів.
   - **Трекери:** Підтримує додавання будь-яких відкритих або закритих трекерів (наприклад: Toloka.to, ThePirateBay, 1337x).

## 🛠 Як запустити (Встановлення)

1. **Підготовка:**
   Впевніться, що у вас встановлено [Docker Desktop](https://www.docker.com/products/docker-desktop/) (якщо ви на Windows) або Docker + Docker Compose (якщо на Linux).
   **Важливо для автозапуску:** У налаштуваннях Docker Desktop (Settings -> General) увімкніть галочку **"Start Docker Desktop when you log in"**. Завдяки параметру `restart: unless-stopped`, сервер буде автоматично запускатися разом із ПК як фонова служба.

2. **Запуск контейнерів:**
   Відкрийте термінал, перейдіть в папку `media-server` і виконайте команду:
   ```bash
   docker compose up -d
   ```

3. **Базове налаштування Jackett:**
   - Відкрийте браузер і перейдіть за адресою: `http://localhost:9117`
   - У верхньому правому кутку скопіюйте ваш **API Key**. Він знадобиться для підключення до Lampa.

4. **Автоматичне налаштування (рекомендовано):**
   Замість того, щоб додавати кожен трекер вручну, скористайтеся скриптом. Він налаштує оптимізацію TorrServer і додасть усі приватні трекери за один крок.
   - Скопіюйте шаблон і впишіть свої дані:
     ```powershell
     Copy-Item .env.example .env   # потім відредагуйте .env
     ```
   - У `.env` вкажіть `JACKETT_APIKEY` (з веб-інтерфейсу Jackett) та логіни/паролі трекерів.
   - Запустіть:
     ```powershell
     ./configure.ps1
     ```
   Скрипт **ідемпотентний** — його можна безпечно запускати повторно. Щоб додати новий приватний трекер, достатньо дописати один рядок у масив `$Trackers` всередині `configure.ps1` та відповідні змінні у `.env`.

> 🔒 **Безпека:** файл `.env` містить паролі й API-ключ і **не потрапляє в git** (додано в `.gitignore`). Ніколи не комітьте його. Публічні трекери (ThePirateBay, 1337x) не потребують логіну — їх можна додати кнопкою **"+ Add indexer"** у веб-інтерфейсі Jackett.

## 📺 Підключення до Lampa

Для того, щоб Lampa побачила ваш новий сервер і почала шукати та відтворювати фільми, виконайте наступні кроки:

### Варіант А: Smart TV (Samsung Tizen) через Media Station X
Якщо у вас телевізор Samsung, найзручніший спосіб запустити Lampa — через безкоштовний додаток-лаунчер:
1. Завантажте **Media Station X** з офіційного магазину Samsung Apps.
2. Запустіть його ➔ **Settings** ➔ **Start Parameter** ➔ **Setup**.
3. Введіть `lampa.mx` та підтвердіть. Додаток перезапуститься і відкриє Lampa.

### Варіант Б: Налаштування самої Lampa (усі пристрої)
1. **TorrServer:** Перейдіть у **Налаштування** -> **Торренти**.
   - Увімкніть використання TorrServer.
   - Вкажіть **Основне посилання**: `http://[IP-АДРЕСА-ВАШОГО-ПК]:8090` (наприклад, `http://192.168.1.165:8090`).
2. **Jackett:** Перейдіть у **Налаштування** -> **Плагіни**.
   - Перейдіть у налаштування плагіна Jackett і вкажіть:
   - **Посилання:** `http://[IP-АДРЕСА-ВАШОГО-ПК]:9117`
   - **API ключ:** Вставте скопійований ключ з вебінтерфейсу Jackett.

## ⚡ Оптимізація швидкості на P2P-рівні

Якщо швидкість низька навіть на торентах із багатьма сідерами — найчастіша причина в тому, що ваш TorrServer **недоступний для вхідних з'єднань** (за NAT). Тоді ви можете лише самі підключатися до відкритих піров, а до вас — ніхто. Це ріже швидкість у рази.

**Рішення — фіксований порт для піров + проброс на роутері:**

1. **Порт вже налаштовано** у цьому репозиторії: `42116` (TCP+UDP) відкрито в `docker-compose.yml` і задано в TorrServer (`PeersListenPort`) через `configure.ps1`. Змінити можна у `.env` (`TORRSERVER_PEER_PORT`).

2. **Проброс на роутері (обов'язково для WAN).** Зайдіть у налаштування роутера → **Port Forwarding / Віртуальний сервер** і додайте правило:
   - Зовнішній порт: `42116`, протокол: **TCP і UDP**
   - Внутрішня IP: IP вашого ПК (напр. `192.168.1.165`), внутрішній порт: `42116`

3. **Брандмауер Windows.** Дозвольте вхідні на `42116` (одноразово, у PowerShell від адміністратора):
   ```powershell
   New-NetFirewallRule -DisplayName "TorrServer P2P" -Direction Inbound `
     -LocalPort 42116 -Protocol TCP -Action Allow
   New-NetFirewallRule -DisplayName "TorrServer P2P UDP" -Direction Inbound `
     -LocalPort 42116 -Protocol UDP -Action Allow
   ```

4. **Перезапуск при зміні порту:** нові порти вимагають перестворення контейнера, не просто рестарту:
   ```powershell
   docker compose up -d torrserver
   ```

**Інші важелі (за потреби):**
- **ISP ріже торенти?** Деякі провайдери сповільнюють P2P. У веб-інтерфейсі TorrServer можна ввімкнути шифрування протоколу (`ForceEncrypt`) — інколи обходить тротлінг (але трохи зменшує пул піров).
- **DHT / PEX / uTP** уже увімкнені — вони дають додаткових піров понад список трекера. Не вимикайте.
- **Головне, що не лікується налаштуваннями:** кількість сідерів на торенті. Обирайте релізи з великим S — це визначає стелю швидкості (див. FAQ).

> Перевірити, що проброс працює: у веб-інтерфейсі TorrServer під час завантаження має рости кількість *активних піров*, а не тільки *pending*.

### Приховати IP через Cloudflare WARP (без платного VPN)

Якщо ви за «сірим» IP (CGNAT) або просто хочете, щоб пири бачили не вашу IP — увесь трафік TorrServer можна завернути через безкоштовний **Cloudflare WARP** (сервіс `warp` на базі [gluetun](https://github.com/qdm12/gluetun) у `docker-compose.yml`). Перевірено: P2P крізь тунель працює, швидкість на добре роздаваних торентах не страждає.

**Що дає / чого не дає:**
- ✅ Пири бачать IP Cloudflare, а не вашу.
- ✅ Може обійти тротлінг BitTorrent з боку провайдера.
- ❌ **Не** прискорює само по собі й **не** дає вхідного порту (ви за NAT Cloudflare) — стелю швидкості визначає кількість сідерів.
- ⚠️ Cloudflare може ріжати P2P — якщо швидкість впаде до нуля, тунель варто вимкнути.

**Налаштування (одноразово):**
1. Згенеруйте WARP-профіль (створює безкоштовний акаунт):
   ```powershell
   docker run --rm -v "${PWD}\warp:/data" -w //data alpine sh -c `
     "apk add --no-cache curl >/dev/null && curl -sL -o wgcf https://github.com/ViRb3/wgcf/releases/download/v2.2.22/wgcf_2.2.22_linux_amd64 && chmod +x wgcf && ./wgcf register --accept-tos && ./wgcf generate && cat wgcf-profile.conf"
   ```
2. З виводу візьміть `PrivateKey` та перший `Address` і впишіть у `.env`:
   ```
   WARP_PRIVATE_KEY=<PrivateKey>
   WARP_ADDRESS_V4=172.16.0.2/32
   ```
3. Підніміть стек: `docker compose up -d`
4. Перевірте, що IP змінився на Cloudflare: `docker exec warp wget -qO- https://api.ipify.org`

**Вимкнути тунель:** у `docker-compose.yml` приберіть `network_mode: "service:warp"` у `torrserver`, поверніть йому власний блок `ports:` (8090 + 42116) і видаліть сервіс `warp`.

> Папка `warp/` (містить ключі акаунта) і `.env` — у `.gitignore`, у git не потрапляють.

## 🎬 Бітрейти та потрібна швидкість (720p → 4K)

Бітрейт — це скільки даних на секунду містить відео. Для стрімінгу через TorrServer він означає одне: **яку швидкість завантаження мають забезпечити сідери, щоб фільм не буферизувався**. Якщо бітрейт релізу вищий за те, що дають сідери — буде підвантаження, хоч би який швидкий був ваш інтернет.

| Якість | Кодек | Оптимальний бітрейт | Потрібно ↓ | Розмір (~2 год) |
|---|---|---|---|---|
| **720p** | H.265 (HEVC) | ~4 Мбіт/с (3–6) | ~0.5 МБ/с | ~4 ГБ |
| **720p** | H.264 | ~6 Мбіт/с (4–8) | ~0.75 МБ/с | ~5 ГБ |
| **1080p** | H.265 (HEVC) | ~8 Мбіт/с (6–12) | ~1 МБ/с | ~7 ГБ |
| **1080p** WEB-DL | H.264 | ~12 Мбіт/с (8–15) | ~1.5 МБ/с | ~11 ГБ |
| **1080p** BluRay | H.264 | ~28 Мбіт/с (20–40) | ~3.5 МБ/с | ~22 ГБ |
| **1440p (2K)** | H.265 (HEVC) | ~16 Мбіт/с (12–24) | ~2 МБ/с | ~14 ГБ |
| **4K (2160p)** WEB-DL | HEVC | ~35 Мбіт/с (25–50) | ~4.5 МБ/с | ~30 ГБ |
| **4K (2160p)** BluRay/UHD | HEVC | ~65 Мбіт/с (50–90) | ~8 МБ/с | ~55 ГБ |
| **4K Remux** | HEVC (без стиснення) | ~100 Мбіт/с (80–120+) | ~12.5 МБ/с | ~80 ГБ |

**Як читати таблицю:**
- Значення — сумарний бітрейт (відео + аудіо), орієнтовний. HDR, 60 fps та динамічні сцени зсувають до верхньої межі.
- **«Потрібно ↓»** — це мінімальна стабільна швидкість завантаження. Вам потрібні **сідери, здатні її тримати**, а не лише швидкий тариф. Наприклад, 4K Remux вимагає ~12 МБ/с — якщо на роздачі 2 сідери по 0.1 МБ/с, він гальмуватиме навіть на каналі 300 Мбіт/с.
- **H.265 (HEVC)** дає ту саму якість удвічі меншим бітрейтом за H.264 — але старі/слабкі пристрої можуть його не тягнути.
- **Золота середина для Smart TV (Samsung/LG):** **1080p WEB-DL H.264** — помірний бітрейт, сумісний звук (AAC), багато сідерів. 4K Remux бери лише коли сідерів сотні й плеєр точно тягне формат.

## 🔍 Моніторинг та логи (PowerShell)

Усі команди працюють з будь-якої папки — Docker знаходить контейнери за іменем (`jackett`, `torrserver`).

```powershell
docker logs jackett --tail 50          # останні 50 рядків
docker logs jackett -f                 # наживо, в реальному часі (Ctrl+C — вийти)
docker logs jackett --since 10m        # тільки за останні 10 хвилин
docker logs jackett -f --timestamps    # наживо + час кожного рядка
```

### Логи TorrServer (відтворення / завантаження)
```powershell
docker logs torrserver --tail 50
docker logs torrserver -f
```

### Фільтрувати логи (аналог grep)
```powershell
docker logs jackett --tail 200 | Select-String "Error"            # тільки помилки
docker logs jackett --tail 200 | Select-String "search"           # тільки пошук
docker logs jackett --tail 200 | Select-String "Error","Found"    # кілька слів
```

### Швидкість і сідери торента без веб-інтерфейсу
```powershell
# список активних торентів + їхні хеші
Invoke-RestMethod -Uri "http://127.0.0.1:8090/torrents" -Method Post `
  -ContentType 'application/json' -Body '{"action":"list"}' | Select-Object title, hash

# швидкість/сідери конкретного торента (підстав свій hash зі списку вище)
$h = "ПІДСТАВ_СВІЙ_HASH"
$t = Invoke-RestMethod -Uri "http://127.0.0.1:8090/torrents" -Method Post `
  -ContentType 'application/json' -Body (@{action='get';hash=$h}|ConvertTo-Json)
"{0} MB/s ↓ | {1} сідерів | {2} піров" -f `
  [math]::Round($t.download_speed/1MB,2), $t.connected_seeders, $t.active_peers
```

### Шпаргалка
| Хочу | Команда |
|---|---|
| Свіжі логи | `docker logs jackett --tail 50` |
| Дивитись наживо | `docker logs jackett -f` |
| Що працює зараз | `docker ps` |

## 💡 Часті питання (FAQ) та вирішення проблем

### 1. Немає звуку під час перегляду (BDRip / Blu-Ray)
- **Причина:** Великі торенти в якості BDRip/Remux використовують багатоканальні аудіоформати (**AC3 / DTS**). Вбудовані плеєри в браузерах, WebOS та Tizen часто не мають ліцензії на їх відтворення, тому відео йде без звуку.
- **Рішення:** 
  - На ПК: Використовуйте зовнішній плеєр (VLC, PotPlayer), вибравши у Lampa опцію "Відкрити у VLC".
  - На Smart TV (Android): Встановіть Vimu Media Player, MX Player або VLC і виберіть його в налаштуваннях Lampa (Налаштування -> Плеєр). 
  - На телевізорах Samsung/LG: Обирайте для перегляду торенти формату **WEB-DL** / **WEBRip** (вони зазвичай мають сумісний звук формату AAC).

### 2. Чому швидкість відображається як 30 МБ/с, якщо у мене тариф 300 Мбіт/с?
- Зверніть увагу на одиниці виміру: провайдери вказують швидкість у Мега**бітах** (Мбіт/с), а Lampa чи TorrServer часто показують у Мега**байтах** (МБ/с, МБайт/с). 
- Оскільки 1 Байт = 8 біт, швидкість 33 МБ/с насправді дорівнює ~265 Мбіт/с. Це означає, що програма працює на максимумі!

### 3. Швидкість завантаження не досягає максимуму, а "стрибає"
- Торент (BitTorrent) — це P2P-мережа. Швидкість залежить від кількості звичайних людей (сідерів), які зараз роздають цей файл, та їхньої швидкості віддачі. 
- Також TorrServer "розумно" знижує швидкість, коли кеш заповнюється, щоб не перевантажувати комп'ютер. Фільм не завантажується повністю, а лише буферизується на льоту.

### 4. Що буде з фільмом, якщо перезавантажити комп'ютер?
- Якщо ви перезавантажите ПК (на якому працює сервер) під час перегляду, фільм на телевізорі зависне. Оскільки TorrServer кешує відео тільки в оперативній пам'яті (щоб не псувати жорсткий диск), після рестарту ПК кеш очиститься.
- **Як продовжити:** Просто увімкніть фільм на телевізорі знову і перемотайте на потрібну хвилину. TorrServer миттєво підключиться до сідерів і продовжить завантаження саме з того моменту (зазвичай це займає 3-5 секунд).

### 5. Фільм раптово зависає, а після перепідключення замість назви — довгий hash
- **Причина:** TorrServer періодично падав через ваду в самому TorrServer MatriX.142 (відкритий баг [YouROK/TorrServer#766](https://github.com/YouROK/TorrServer/issues/766) — гонитва даних у torrent-бібліотеці під паралельним навантаженням). Docker автоматично перезапускає сервіс за ~5 секунд; поки Lampa наново тягне метадані торента, вона показує його infohash — це тимчасово.
- **Рішення (вже застосовано в цьому репозиторії):** версію TorrServer прикріплено до стабільної `MatriX.141.10` через образ `ghcr.io/yourok/torrserver` (тег задається змінною `TORRSERVER_VERSION` у `.env`). Образ `yourok/torrserver:latest` **не використовуйте** — він при кожному старті завантажує найновіший бінарник із GitHub, тобто версія неконтрольована.
- Коли баг #766 виправлять — просто підніміть `TORRSERVER_VERSION` у `.env` і виконайте `docker compose up -d torrserver`.
- Перевірити поточну версію: `Invoke-RestMethod http://127.0.0.1:8090/echo`. Дані/налаштування тепер зберігаються у `torrserver_data/` і переживають перестворення контейнера.

### 6. Lampa на телевізорі пише «парсер не відповідає», хоча з телефону сторінка `:9117` відкривається
Це майже завжди одна з двох причин (сторінка відкривається з телефону, бо це звичайна навігація браузера, а Lampa робить фонові JS-запити, до яких застосовуються інші правила безпеки):

1. **CORS вимкнено в Jackett.** Браузер блокує відповідь Jackett, бо запит іде з чужого домену (`lampa.mx`).
   - **Рішення:** у файлі `jackett_config/Jackett/ServerConfig.json` встановіть `"AllowCORS": true` і перезапустіть контейнер: `docker restart jackett`.
2. **Mixed content (HTTPS → HTTP).** Якщо Lampa відкрита по `https://`, а Jackett працює по `http://`, браузер телевізора блокує «незахищений» запит.
   - **Рішення:** у Media Station X у полі *Start Parameter* вкажіть Lampa через **http** (`http://lampa.mx`), щоб і Lampa, і Jackett були по HTTP.

Також перевірте, що:
- В Lampa вказано саме LAN-IP ПК, а не `localhost`: `http://192.168.1.165:9117` (свій IP: `ipconfig` → рядок IPv4).
- Телевізор і ПК в **одній мережі** (без «гостьового» Wi-Fi чи AP-isolation).
- Брандмауер Windows пропускає вхідні на порт `9117` для приватної мережі.
